# Batch Size Sweet Spot Record: PD + Speculative Decoding

日期：2026-04-14

## 目的

记录关于批量大小/并发最佳点的一项具体观察
比较推测解码时在“nvidia/Kimi-K2.5-NVFP4”上服务的 PD
相对于非投机基线。

本说明基于两次守夜活动：

- `third_partys/vigil/logs/pd_dev/2026-04-14/20260414_074458`
- `third_partys/vigil/logs/pd_dev/2026-04-14/20260414_082518`

## 设置

两次运行的共享设置：

- 型号：`nvidia/Kimi-K2.5-NVFP4`
- 拓扑：PD分解，“1P x 4 GPU + 1D x 4 GPU”
- 预填充：V1 跑步者，渴望
- 解码：使用 `FULL_DECODE_ONLY` cudagraph 的 V2 风格配置
- KV 传输：解码时“NixlConnector”，预填充时“NixlConnector + SimpleCPUOffloadConnector”
- 基准测试：`vllm-bench`，ShareGPT 合成多轮工作负载
- 扫：多回合并发`4, 8`

主要区别：

- `20260414_074458`：使用 Eagle3、3 个草稿令牌启用推测解码
- `20260414_082518`：推测解码已禁用

证据：

- 规格运行重现文件：
  `third_partys/vigil/logs/pd_dev/2026-04-14/20260414_074458/REPRODUCE.md`
- 非规格运行重现文件：
  `third_partys/vigil/logs/pd_dev/2026-04-14/20260414_082518/REPRODUCE.md`

## 主要结果

对于此 PD 设置，最佳点因并发性而异：

- 在较低的批量压力（`并发= 4`）下，推测解码得到改善
  输出吞吐量和 TPOT，但会损害 TTFT。
- 在较高的批量压力（`并发= 8`）下，推测性解码更差
  高于吞吐量、TTFT、TPOT 和 E2EL 的非规格基准。

简而言之：

- `concurrency=4` 仍然是一个可行的猜测区域。
- `concurrency=8` 超出了这个推测性配置的最佳范围。## 基准测试总结

- 规格平均 E2EL 恶化约“23%”

## 为什么会发生这种情况

在支持规范的运行中，推测路径是活跃且健康的。解码日志
显示 Eagle3 初始化和实时接受指标：

- 加载Eagle3辅助层
- Eagle 投机者执行 CUDA 图形捕获
- 平均草稿接受率约为“59%”至“61%”
- 平均接受长度约为“2.76”至“2.86”

这意味着猜测并未被打破。它有助于降低并发性，因为
解码端可以更快地生成令牌。但在并发数较高的情况下，额外的
在此 PD 设置中，草稿/验证工作不再物有所值，因此端到端
延迟和吞吐量均有所下降。相关日志示例：

- `third_partys/vigil/logs/pd_dev/2026-04-14/20260414_074458/decode-0/decode-0-gb200-rack1-04.log`
- `third_partys/vigil/logs/pd_dev/2026-04-14/20260414_082518/decode-0/decode-0-gb200-rack1-08.log`

## 甜蜜点外卖

对于这个确切的配置：

- 如果针对中等并发性进行优化，推测解码可能会很有用。
- 如果操作点更接近“concurrency=8”，则非规范基线为
  更好的选择。

从操作上来说，当前投机的最佳点似乎低于
`concurrency=8`，其中 `concurrency=4` 显示出积极的权衡
`concurrency=8` 显示出明显的回归。

## 警告

这不是一个完美的节点控制 A/B 测试：

- 规格运行使用“gb200-rack1-03”和“gb200-rack1-04”
- 非规格运行使用“gb200-rack1-07”和“gb200-rack1-08”

因此节点方差可能会导致差异。尽管如此，观察到的趋势是
足够一致，可以将其视为批量大小的强有力的工作假设
甜蜜点选择。
