# Simple Top-Down System Design for vLLM + MooncakeStoreConnector

## 1. 目标

设计一个简单的 V1 集成，以便在多节点 GB200/Grace 集群上的 vLLM 中使用“MooncakeStoreConnector”。

V1优先级：

- 性能第一。
- 操作简单性其次。
- 如果可以消除主要数据路径瓶颈，则较高的复杂性是可以接受的。

本文档提出了**节点范围的 Mooncake 所有权模型**：

- 每个集群一台 Mooncake **主服务器**
- 每个节点一个 Mooncake **所有者客户端**
- 每个节点有多个 vLLM 本地 **虚拟客户端**

这有意不同于纯粹的“每个员工一个商店客户”的模式。主要原因是在节点级别集中 RDMA 所有权、NIC 绑定和主机内存管理。

## 2. 设计假设

- “所有者客户端”映射到节点范围的 Mooncake `RealClient` 服务。
- “虚拟客户端”映射到每个工人的 Mooncake `DummyClient` 前端，将请求转发给本地所有者。
- GB200 节点具有**两个 Grace CPU**，因此我们将该节点视为**两个 NUMA/位置域**。
- 所有者客户端可以在节点上绑定**四个 RDMA NIC**，并向所有本地工作人员公开一项共享传输服务。
- 如果节点的 RDMA 堆栈发生故障，V1 接受该节点作为故障域。

```
   - Grace 0 <-> GPU0/GPU1

```
## 5. 端到端数据流

### 保存/卸载路径

1. vLLM 工作人员决定卸载 KV 块。
2.worker 调用其本地虚拟客户端。
3. 虚拟客户端将请求转发给节点所有者客户端。
4. 所有者客户端向集群主机请求放置。
5. 所有者客户端对所选内存段执行RDMA写入。

目标放置可能是：

- 本地节点CPU内存
- 远程节点CPU内存

### 加载/重用路径

1. vLLM工作人员需要KV块。
2.worker 调用其本地虚拟客户端。
3. 所有者客户端通过主元数据解析副本。
4. 所有者客户端执行 RDMA 读取目标 GPU 缓冲区。

V1政策：

- 当元数据允许时更喜欢最短的本地路径
- 否则直接从远程节点内存读取
- 在主路径稳定之前不要针对每个极端情况进行优化

## 6. 关键设计选择

### 6.1 每个节点一个所有者客户端

为什么：

- 避免每个工作人员重复 RDMA 设置
- 让我们将所有节点网卡绑定在一处
- 为每个节点提供一个共享 CPU 内存池
- 使 NUMA 感知布局可实施

### 6.2 RDMA 在节点处终止，而不是在每个工作线程处终止

所有 RDMA 资源应由节点范围的所有者客户端拥有。

这保留了：

- 网卡绑定
- 队列对生命周期
- 记忆登记
- 重试和错误处理

将它们集中在一处，而不是将它们分散到所有工作进程中。

### 6.3 NUMA 感知数据路径

所有者客户端应分配具有 NUMA 感知的 CPU 全局段，并将首选路径与最近的 Grace 域和 RDMA NIC 对齐。

实用意图：

- GPU0/GPU1 更喜欢 Grace0-本地 CPU 内存
- GPU2/GPU3 更喜欢 Grace1-本地 CPU 内存
- 按 NIC NUMA 域分段有助于高效使用所有四个 NIC### 6.4 共享缓冲和复制模型

如果需要，V1 接受**一份额外的节点本地副本**到所有者可见的共享内存中，但避免为每个工作客户端支付一份额外的副本。

所以模型是：

- 所有者级别的共享暂存/缓冲区管理
- 每个虚拟客户端内没有单独的完整数据路径复制堆栈

### 6.5 两阶段分配，本地共享

有两个分配层：

1. 主服务器进行集群分配，用于最终的月饼副本放置
2. 所有者客户端为共享传输/暂存资源进行节点本地分配

第二层由所有本地虚拟客户端共享。

## 7. 故障模型

V1 有意使用粗略故障域：

- 如果所有者客户端或节点级 RDMA 路径发生故障，则整个节点实例被视为不可用
- 虚拟客户端是轻量级前端，而不是独立的恢复域

这在 V1 中是可以接受的，因为它降低了控制平面的复杂性并保持快速路径的简单。

## 8. 为什么这是一个好的 V1

- 它与 Mooncake 现有的 `RealClient` / `DummyClient` 拆分很好地匹配。
- 它提供了一个干净的地方来实现 NUMA 感知的 CPU 段安装。
- 它允许一个所有者绑定多个 RDMA NIC。
- 它使 vLLM 方面的工作人员集成保持简单。
- 以后很容易向更智能的位置感知分配和更细粒度的故障转移发展。

## 9. 超出 V1 范围

- 细粒度的每个工作人员故障转移
- SSD卸载策略重新设计
- 跨拓扑层的高级副本评分
- 完整的调度程序更改以实现完美的位置放置

## 10. 总结

推荐的V1设计是：

- **一个集群，一台主服务器**
- **一个节点，一个所有者客户端**
- **一名工人，一名虚拟客户**
- **节点所有者拥有的所有 RDMA**
- **跨越两个 Grace 域的 NUMA 感知 CPU 段**这为我们提供了一个简单的自上而下的架构，该架构足够快，值得构建，同时保持 vLLM 工作线程和 Mooncake 节点服务之间​​清晰的实现边界。
